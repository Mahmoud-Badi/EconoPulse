# Service Hub Generator

**Purpose:** Create a single-source-of-truth hub file for each service by consolidating
specs, screens, API endpoints, and component catalogs into one navigable document.

**Output:** One hub file per service in `dev_docs/specs/services/{service}.md` + index file

---

## Mode Selection

> **Choose the mode BEFORE running this generator.** The mode determines which inputs are required and which generation steps execute.

### `MODE: greenfield`

Use when building a new project from scratch (no existing codebase). Hub files are populated entirely from service specs and tribunal output. No code scanning occurs.

**Required inputs (greenfield):**

| Input | Location | Required? | What it provides |
| ----- | -------- | --------- | ---------------- |
| Service Spec | `dev_docs/specs/services/{service}.md` or tribunal specs | **YES** | Business rules, workflows, data model |
| Tribunal Output | `{{TRIBUNAL_OUTPUT_PATH}}/` | **YES** | Feature scope, priorities, implementation details |
| Screen Catalog | `{{SCREEN_CATALOG_PATH}}` | Optional | If screen specs have been generated |
| Component Catalog | `dev_docs/components/_index.md` | No | Does not exist yet in greenfield |
| API Documentation | N/A | No | No code exists to scan |
| Audit Report | N/A | No | No code exists to audit |

**Greenfield generation rules:**

1. **API Endpoints section:** Populate entirely from the service spec's endpoint definitions. Mark all endpoints as `Status: Not Built`.
2. **Screens section:** If screen specs exist, pull from those. Otherwise, infer screens from the service spec's workflow descriptions and mark all as `Status: Not Built`.
3. **Components Used section:** Leave empty with note: `Components will be cataloged after implementation begins.`
4. **Implementation Status section:** Set ALL layers to `Not Built` with notes describing what needs to be created.
5. **Known Issues section:** Leave empty — no code means no bugs. Add note: `No known issues — project has not entered development.`
6. **Quality Grade:** Set to `N/A` (not yet implemented).
7. **Tasks section:** Pull task IDs from `dev_docs/tasks/` if PHASE-GENERATOR has run. Otherwise, leave with note: `Tasks will be generated by PHASE-GENERATOR.`

### `MODE: existing`

Use when working with an existing codebase. Hub files are populated by scanning code AND reading specs, then merging findings.

**Required inputs (existing):**

| Input | Location | Required? | What it provides |
| ----- | -------- | --------- | ---------------- |
| Service Spec | `dev_docs/specs/services/{service}.md` or tribunal specs | **YES** | Business rules, workflows, data model |
| Screen Catalog | `{{SCREEN_CATALOG_PATH}}` | **YES** | Screens belonging to this service |
| API Documentation | Backend controller files or Swagger/OpenAPI | **YES** | Endpoints, DTOs, auth requirements |
| Component Catalog | `dev_docs/components/_index.md` | **YES** | Components used by this service |
| Audit Report | `dev_docs/audit/{service}-audit.md` | **YES** | Quality grade, known bugs |

**Existing codebase generation rules:**

1. **API Endpoints section:** Scan backend controllers for route decorators AND read service spec endpoints. Merge both lists. Mark discrepancies (endpoint in code but not spec, or in spec but not code).
2. **Screens section:** Scan frontend route directory AND read screen catalog. Flag screens that exist in code but not in specs (undocumented) and screens in specs but not in code (not yet built).
3. **Components Used section:** Scan component files used by this service's screens. Cross-reference with component catalog.
4. **Implementation Status section:** Check actual file existence for each layer (Prisma models, controllers, pages, tests).
5. **Known Issues section:** Pull from audit report's bug inventory.
6. **Quality Grade:** Pull from audit report's quality assessment.

### Mode Detection

If mode is not explicitly set, detect it automatically:

- If `dev_docs/audit/` directory contains audit reports → `MODE: existing`
- If no audit reports AND no backend controller directory exists → `MODE: greenfield`
- If ambiguous → **ASK** the user: "Is this a greenfield project or an existing codebase?"

---

## When to Run

Run this generator after:

1. Service specifications exist (from tribunal or original docs)
2. **(MODE: existing only)** Backend endpoints are at least partially implemented
3. **(MODE: existing only)** Screen catalog entries exist for the service
4. **(MODE: greenfield)** Tribunal process is complete and service specs are generated

## Generation Algorithm

1. **Read the service specification.** Extract:
   - Service description and purpose
   - Business rules and constraints
   - Data model (entities, fields, relations)
   - Workflow steps

2. **Scan backend controllers** for this service. Extract:
   - All route decorators (`@Get`, `@Post`, `@Patch`, `@Delete`)
   - Controller method names
   - DTO class names (request/response shapes)
   - Auth guards (`@UseGuards`, `@Public`)

3. **Read the screen catalog.** Filter for screens belonging to this service:
   - Match by route prefix (e.g., `/loads/*` belongs to Load service)
   - Match by component directory (e.g., `components/tms/loads/`)

4. **Read the component catalog.** Filter for components used by this service:
   - Domain-specific components in the service directory
   - Shared components referenced by this service's screens

5. **Read the audit report** for this service. Extract:
   - Quality grade
   - Bug list with severities
   - Recommendations

6. **Consolidate** all extracted data into the hub file format above.

7. **Track implementation status** per layer:
   - Database: Check if Prisma models exist and have migrations
   - API: Check if controllers have real logic (not just stubs)
   - Frontend: Check if screens render real data (not mock data)
   - Tests: Check if test files exist and pass

---

## Quality Rules

1. **One hub per service.** If a service has sub-modules, they go in the same hub as sections.
2. **Every endpoint must be listed.** Even stubs -- mark them with status "Stub".
3. **Every screen must have a route and task ID.** No orphan screens.
4. **Business rules are non-negotiable.** These are the "you MUST know this" items.
5. **Hub files are living documents.** Update them as tasks complete.
6. **Cross-reference task IDs.** Every known issue links to a task. Every planned screen links to a task.

---

## Validation Checklist

After generation, verify:

- [ ] Every endpoint in the controller appears in the hub's API table
- [ ] Every screen in the app router for this service appears in the Screens table
- [ ] Every bug from the audit report appears in Known Issues
- [ ] Implementation Status accurately reflects current state (check file existence)
- [ ] Business rules match the service specification
- [ ] Links to specs, audits, and tasks are valid paths
- [ ] Service index file is updated with this service entry
