## 1. Migration Types

| Type | When to Use | Risk Level | Rollback Difficulty |
|------|------------|------------|-------------------|
| **Schema migration** | Add/alter/drop columns, tables, indexes | Medium | Easy (reverse SQL) |
| **Data migration** | Transform existing data, backfill columns | High | Hard (data changes) |
| **Zero-downtime migration** | Production changes with live traffic | High | Medium (expand-contract) |
| **Seed migration** | Initialize reference/lookup data | Low | Easy (re-run seed) |

## 2. Migration Tooling

<!-- IF {{ORM}} == "prisma" -->
### Prisma Migrations

```bash
# Generate migration from schema changes
npx prisma migrate dev --name descriptive-name

# Apply pending migrations
npx prisma migrate deploy

# Reset database (DESTRUCTIVE — dev only)
npx prisma migrate reset

# Check migration status
npx prisma migrate status
```

**Key rules:**
- Never edit generated SQL migration files after they've been committed
- Always review the generated SQL before committing: `prisma/migrations/*/migration.sql`
- Use `prisma db push` only for prototyping — never in production pipeline
- Run `npx prisma generate` after every schema change to update the client
<!-- ENDIF -->

<!-- IF {{ORM}} == "drizzle" -->
### Drizzle Migrations

```bash
# Generate migration SQL from schema changes
npx drizzle-kit generate

# Apply pending migrations
npx drizzle-kit migrate

# Push schema directly (dev only, no migration file)
npx drizzle-kit push
```

**Key rules:**
- Always use `generate` not `push` for any change that needs to be reproducible
- If using `pgSchema`, `push` has known bugs with enum qualification — use `generate` always
- Review generated SQL in `drizzle/` directory before committing
- Enum string literals need `as const` to prevent type widening (see LESSONS-LEARNED.md)
<!-- ENDIF -->

<!-- IF {{ORM}} == "sqlalchemy" -->
### Alembic Migrations (SQLAlchemy)

```bash
# Generate migration from model changes
alembic revision --autogenerate -m "descriptive-name"

# Apply pending migrations
alembic upgrade head

# Rollback one step
alembic downgrade -1

# Check current version
alembic current
```

**Key rules:**
- Always review autogenerated migrations — Alembic cannot detect all changes (e.g., column renames)
- Include both `upgrade()` and `downgrade()` functions in every migration
- Test migrations on a copy of production data before deploying
<!-- ENDIF -->

<!-- IF {{ORM}} == "activerecord" -->
### Rails Migrations (ActiveRecord)

```bash
# Generate migration
rails generate migration DescriptiveName

# Apply pending migrations
rails db:migrate

# Rollback one step
rails db:rollback

# Check migration status
rails db:migrate:status
```

**Key rules:**
- Always include a `reversible` block or implement both `up` and `down` methods
- Use `change` method only for reversible operations (add_column, create_table, add_index)
- For data migrations, use a separate rake task — not a Rails migration
<!-- ENDIF -->

## 3. Migration Workflow

Every schema change follows this pipeline:

```
1. Local Development
   └── Modify schema/models
   └── Generate migration file
   └── Review generated SQL
   └── Run migration locally
   └── Update seed data if needed
   └── Run full test suite
   └── Commit migration file + schema changes together

2. Code Review
   └── Reviewer checks migration SQL for:
       - Destructive operations (DROP, DELETE, TRUNCATE)
       - Missing indexes on foreign keys
       - Missing NOT NULL defaults for existing data
       - Potential lock contention on large tables

3. Staging Deployment
   └── Apply migration to staging database
   └── Verify application works with new schema
   └── Run integration tests against staging

4. Production Deployment
   └── Take database backup (automated pre-deploy)
   └── Apply migration during low-traffic window
   └── Monitor error rates for 15 minutes post-deploy
   └── Verify rollback plan is executable
```

## 4. Rollback Strategy

### Before Every Migration

- [ ] Database backup is current (< 1 hour old)
- [ ] Rollback SQL exists and has been tested
- [ ] Application code is backward-compatible with old schema (if zero-downtime)
- [ ] Team knows the rollback procedure

### Rollback Decision Matrix

| Scenario | Action |
|----------|--------|
| Migration fails mid-execution | Automatic rollback (transaction-wrapped) |
| App errors spike after migration | Manual rollback within 15 minutes |
| Data corruption detected | Restore from backup + investigate |
| Performance degradation | Monitor for 1 hour, then decide |

## 5. Data Backfill Patterns

When adding a new NOT NULL column to a table with existing data:

### The Expand-Contract Pattern

```
Step 1 (Expand):  Add column as NULLABLE with default
Step 2 (Backfill): Populate column for existing rows (batch job)
Step 3 (Contract): Add NOT NULL constraint after backfill completes
```

### Batch Processing Template

```sql
-- Backfill in batches of 1000 to avoid locking
DO $$
DECLARE
  batch_size INT := 1000;
  affected INT;
BEGIN
  LOOP
    UPDATE {{TABLE_NAME}}
    SET new_column = computed_value
    WHERE new_column IS NULL
    LIMIT batch_size;

    GET DIAGNOSTICS affected = ROW_COUNT;
    EXIT WHEN affected = 0;

    RAISE NOTICE 'Updated % rows', affected;
    PERFORM pg_sleep(0.1);  -- Brief pause to reduce lock contention
  END LOOP;
END $$;
```

### Backfill Rules

- Always process in batches (1000-5000 rows per batch)
- Make backfill scripts idempotent — safe to re-run
- Log progress so you can monitor and estimate completion time
- Run during low-traffic hours for large tables
- Test on a staging copy of production data first

## 6. Zero-Downtime Migration Checklist

For any migration that runs against a live production database:

- [ ] **Additive only** — new columns/tables only, no drops or renames
- [ ] **New columns are nullable** or have a server-side default
- [ ] **Application code handles both old and new schema** (deploy code first, then migrate)
- [ ] **No exclusive table locks** — avoid `ALTER TABLE ... ADD CONSTRAINT` on large tables without `CONCURRENTLY`
- [ ] **Indexes created concurrently** — use `CREATE INDEX CONCURRENTLY` to avoid blocking writes
- [ ] **Tested with production-scale data** — a migration that takes 2ms on 100 rows may take 20 minutes on 10M rows

### Column Rename (Zero-Downtime)

Never rename a column directly. Instead:

```
1. Add new column (nullable)
2. Deploy code that writes to BOTH old and new columns
3. Backfill new column from old column
4. Deploy code that reads from new column only
5. Drop old column (next release cycle)
```

### Table Rename (Zero-Downtime)

Never rename a table directly. Instead:

```
1. Create new table
2. Set up trigger to sync writes: old → new
3. Backfill new table from old table
4. Deploy code that reads/writes new table
5. Remove trigger and drop old table (next release cycle)
```

## 7. Migration Naming Convention

```
YYYYMMDDHHMMSS_descriptive_action.sql
```

Examples:
- `20260301120000_add_status_column_to_projects.sql`
- `20260305143000_create_task_comments_table.sql`
- `20260310090000_add_index_on_tasks_project_id.sql`
- `20260315160000_backfill_project_slug_from_name.sql`

**Rules:**
- Timestamp prefix ensures execution order
- Descriptive name tells you what it does without reading the SQL
- One logical change per migration file
- Never reuse or edit a committed migration's timestamp
