# Hub File Correction Checklist

## Purpose

After a Per-Service Tribunal produces a MODIFY or REVERSE verdict, the hub file must be corrected to reflect reality. This 15-step checklist ensures no section is missed and all corrections are verified.

**Critical lesson from Ultra TMS:** Hubs written from specs had a 97% error rate. Hubs corrected from code inspection are accurate. Always verify against code, never against specs.

---

## Pre-Correction Setup

- [ ] PST file for this service is complete and reviewed
- [ ] User has approved the tribunal verdict
- [ ] You have the hub file open AND access to the codebase
- [ ] You are in a fresh chat session (prevents context contamination)

---

## The 15-Step Checklist

### Header & Status (Steps 1-3)

- [ ] **Step 1: Update Status Box** — Set health score to PST verified score, update confidence level, set "Last Verified" to today's date, update backend/frontend/test status badges

- [ ] **Step 2: Update Implementation Status Matrix** — Set each layer (design, backend, models, frontend, hooks, components, tests, security) to actual verified status, not planned status

- [ ] **Step 3: Add Tribunal Link** — Add line: `> **Tribunal:** [PST-XX](path/to/PST-XX-service-name.md) | Verdict: MODIFY | Score: X/10 → Y/10`

### Content Sections (Steps 4-10)

- [ ] **Step 4: Fix Endpoint List** — Remove phantom endpoints (listed but don't exist in code). Add missing endpoints (exist in code but not listed). Update status of each endpoint (Production/Stub/Not Built). Verify method, path, and controller for each.

- [ ] **Step 5: Fix Screen List** — Remove phantom routes. Add missing routes. Update quality scores to PST-verified scores. Update LOC counts if significantly wrong.

- [ ] **Step 6: Fix Component Count** — Update component count to match `find . -name "*.tsx" | wc -l` in the service's component directory. List all components, remove phantom ones.

- [ ] **Step 7: Fix Hook Count** — Update hook count. List all hooks with their purpose. Remove phantom hooks.

- [ ] **Step 8: Fix Data Model** — Correct model names, field names, field types, relations. Remove phantom models/fields. Add models that exist in schema but weren't documented. Fix enum values.

- [ ] **Step 9: Fix Test Section** — Update test file count (backend unit, frontend unit, integration, E2E). Update test quality tier (Empty Shell / Basic Assertions / Comprehensive). Remove false "No tests" or "None" claims. List actual test files with paths.

- [ ] **Step 10: Fix Business Rules** — Verify each listed rule is actually implemented in code. Remove rules that are documented but not implemented (move to backlog). Add rules that are implemented but not documented.

### Cross-References & Metadata (Steps 11-13)

- [ ] **Step 11: Fix Dependencies** — List actual runtime dependencies (other services, external APIs, databases). Remove phantom dependencies.

- [ ] **Step 12: Fix Known Issues** — Close issues that have been fixed (strikethrough + fix date). Add new issues found during PST. Update priority of existing issues.

- [ ] **Step 13: Update Task Links** — Link to active tasks. Remove links to completed tasks. Add links to new tasks generated by PST.

### Verification (Steps 14-15)

- [ ] **Step 14: Three-Way Verification** — Compare corrected hub against PST findings AND code. All three must agree. Any discrepancy = re-check.

- [ ] **Step 15: Format Conformance** — Verify hub follows the standard 15-section template. All sections present, correct order, correct markdown formatting. No duplicate sections.

---

## Post-Correction

- [ ] Mark this service as "CORRECTIONS APPLIED" in `_PROGRESS.md`
- [ ] Update aggregate metrics (mean hub score should improve)
- [ ] If corrections revealed new findings → add to CCF tracker
- [ ] Commit hub file with message: `docs: correct {service} hub per PST-XX tribunal verdict`

---

## Verification Tests (Automated)

Run these checks on the corrected hub:

1. **No phantom endpoints:** Every endpoint listed in hub exists in code (`grep -r "@Get\|@Post" ...`)
2. **No phantom components:** Every component listed exists as a file
3. **No "None" or "No tests":** Search hub for false negatives
4. **Section count:** Hub has all 15 sections
5. **Tribunal link present:** Hub references its PST file
6. **Score consistency:** Hub health score matches PST verified score
7. **Date freshness:** "Last Verified" date is today or this week
