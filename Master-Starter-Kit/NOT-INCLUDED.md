# What This Kit Is (and Isn't)

> Read this before buying or starting, so you know exactly what you're getting.

---

## What This Kit IS

**A project planning accelerator.** You describe your idea, and an AI agent generates a complete project plan — architecture, specs, tasks, design system, and research — in hours instead of weeks.

Think of it like a **senior technical co-founder** who:
- Asks the right questions about your product
- Researches your competitors
- Designs your architecture
- Writes every spec and task file
- Hands you a organized plan and says "start building"

---

## What This Kit is NOT

### It's not a code generator
The kit produces **documentation and specs**, not application code. After running the kit, you still need to write code (or use an AI coding assistant) to build the actual product. The kit tells you **what** to build and **in what order** — not how to write the code.

### It's not a no-code tool
You can't drag and drop your way to a finished product. The kit requires Claude Code (an AI coding tool) to run, and the output is markdown files and planning documents — not a deployed application.

### It's not a project management tool
The kit generates a STATUS.md sprint dashboard and task files, but it doesn't replace tools like Jira, Linear, Notion, or Trello. You'll execute the plan in whatever project management tool you prefer. The kit creates the plan; you manage the execution.

### It's not a replacement for domain expertise
The kit asks smart questions and generates thorough specs, but it needs **your knowledge** of your industry, your users, and your market. If you're building a healthcare app, the kit won't teach you HIPAA compliance — but it will ask you "does this app handle protected health information?" and generate the right architecture if you say yes.

### It's not one-size-fits-all
A personal blog doesn't need 30 steps of planning. A multi-tenant SaaS does. That's why the kit has [multiple paths](PATHS.md) — pick the one that fits your project. Using the full pipeline for a small project is overkill. Using the lite path for a funded startup is underkill.

### It's not magic
The kit is only as good as the information you give it. If you describe your project vaguely, you'll get vague specs. If you describe it precisely, you'll get precise specs. The 27 intake questions exist to extract precision even if you start vague.

## What You'll Need After Running the Kit

| What | Why |
|------|-----|
| **Coding ability** (yours or AI-assisted) | To build the actual product from the specs |
| **A hosting platform** (Vercel, AWS, etc.) | To deploy the finished product |
| **A database** (PostgreSQL, MySQL, etc.) | To store your data (the kit designs the schema, you provision the database) |
| **Time** | The kit saves weeks of planning, but you still need to write and ship code |

---

## What You WON'T Need After Running the Kit

| What | Why Not |
|------|---------|
| **An architect** | The kit produces architecture docs |
| **A product manager** (for planning) | The kit produces feature prioritization and phase plans |
| **A competitor analyst** | The Tribunal produces competitive research |
| **A technical writer** (for specs) | The kit produces service specs, screen specs, and API contracts |
| **Weeks of "what should we build?"** | The verdict document answers this question definitively |

## The Honest Sales Pitch

If you're building a software product and you've ever:
- Started coding without a plan, then regretted it
- Redesigned your database after week 3
- Argued with your team about what features to build first
- Lost context between coding sessions
- Shipped features nobody asked for

...this kit prevents all of that. It's 4 hours of planning that saves weeks of rework.

If you're just experimenting, learning to code, or building something tiny — the [Lite Path](PATHS.md) is enough. You don't need the full pipeline.
